============================================================
HOW THE CODE WORKS - FULL EXPLANATION
============================================================

OVERVIEW
--------
This is a real-time voice-to-AI app. You speak into your browser mic,
your speech is transcribed live by Deepgram, and then Groq AI replies
to what you said. Everything happens over WebSockets so it feels instant.


TOOLS USED
----------
- FastAPI     : Python web server
- Deepgram    : Converts your voice to text (Speech-to-Text)
- Groq        : AI that replies to your spoken text (LLM)
- WebSocket   : A live two-way connection between browser and server
- MediaRecorder (browser) : Records your mic and sends audio chunks


------------------------------------------------------------
STEP BY STEP - WHAT HAPPENS WHEN YOU OPEN THE PAGE
------------------------------------------------------------

STEP 1 - You open http://localhost:8000
  The server sends back the HTML page with the JavaScript already in it.
  The page has two sections:
    - "What you spoke"  -> shows your transcribed speech
    - "AI Response"     -> shows the AI reply to what you said


STEP 2 - Browser asks for mic permission
  The JavaScript calls navigator.mediaDevices.getUserMedia({ audio: true })
  This asks your browser for access to your microphone.


STEP 3 - Browser connects to the server via WebSocket
  A WebSocket connection opens to ws://localhost:8000/ws
  This is a live two-way channel that stays open the whole time.
  When it connects, the page shows "Connected".


STEP 4 - Browser starts recording your mic
  MediaRecorder starts recording audio in webm format.
  Every 250 milliseconds it sends a small audio chunk to the server
  through the WebSocket. This is continuous as long as the page is open.


STEP 5 - Server receives audio and sends it to Deepgram
  The server has a simple loop:
    while True:
        data = receive audio chunk from browser
        send it to Deepgram

  This loop runs non-stop and is never blocked.
  Deepgram receives a continuous stream of audio from your mic.


STEP 6 - Deepgram listens and transcribes
  Deepgram processes the audio stream in real time.
  When it detects a complete sentence or phrase, it sends back a transcript.
  Example: you say "What is Python?" -> Deepgram sends back "What is Python?"


STEP 7 - Server receives the transcript and shows it
  A background task is running alongside the audio loop.
  This task listens for messages coming back from Deepgram.
  When a transcript arrives, the server:
    1. Sends the transcript text to the browser -> appears in "What you spoke"
    2. Sends that same text to Groq AI as a question


STEP 8 - Groq AI generates a reply
  Groq uses the llama-3.3-70b model to reply to what you said.
  The reply is streamed back chunk by chunk (word by word).
  Each word chunk is sent to the browser prefixed with "ai:"
  Example sent to browser:  "ai:Python"  "ai: is"  "ai: a"  "ai: programming"


STEP 9 - Browser displays the AI reply
  The browser checks every message it receives:
    - If it starts with "ai:" -> strip the prefix, add to "AI Response" section
    - If it does NOT start with "ai:" -> add to "What you spoke" section

  This is how two different outputs go to two different sections on the page.


------------------------------------------------------------
WHAT WAS BROKEN IN THE ORIGINAL CODE AND WHAT WE FIXED
------------------------------------------------------------

PROBLEM 1 - Wrong SDK version
  Original code used:  from deepgram import Deepgram
  That import was from Deepgram SDK version 1 (very old).
  The installed version is 7.3.1 which is completely different.
  In 7.3.1, the class "Deepgram" does not exist at all.

  Fix: Changed to:  from deepgram import AsyncDeepgramClient
  This is the correct class in version 7.3.1 for async WebSocket use.


PROBLEM 2 - Wrong way to open the Deepgram connection
  Original code used:
    connection = await dg_client.transcription.live({'punctuate': True})
  That method does not exist in version 7.3.1.

  Fix: Changed to:
    async with dg_client.listen.v1.connect(model="nova-2", punctuate=True) as deepgram_socket:
  This is the correct way in 7.3.1. It also automatically closes the
  connection when done, which the old code never did.


PROBLEM 3 - Wrong way to listen for transcripts
  Original code used event callbacks:
    connection.registerHandler(connection.event.TRANSCRIPT_RECEIVED, handler)
  That method does not exist in version 7.3.1.

  Fix: In 7.3.1, you receive messages by iterating the socket:
    async for message in deepgram_socket:
        if isinstance(message, ListenV1Results):
            transcript = message.channel.alternatives[0].transcript


PROBLEM 4 - Transcript was a dict in old SDK, now it is a typed object
  Original code accessed transcript like a dictionary:
    data['channel']['alternatives'][0]['transcript']
  In version 7.3.1, the result is a Pydantic object, not a raw dict.

  Fix: Access it using dot notation:
    message.channel.alternatives[0].transcript


PROBLEM 5 - Missing "await" on send
  Original code:  deepgram_socket.send(data)
  This was missing "await", so the audio was never actually sent.
  Also, "send()" does not exist in 7.3.1.

  Fix:  await deepgram_socket.send_media(data)
  The method is renamed and must be awaited.


PROBLEM 6 - Groq was blocking the audio loop (BIGGEST BUG - caused empty transcript)
  Original code called Groq INSIDE the audio send loop:
    while True:
        data = receive audio chunk
        send to Deepgram
        CALL GROQ AND WAIT FOR FULL RESPONSE  <-- this blocked here for 5-10 seconds

  Because Groq takes several seconds to respond, the loop was stuck.
  Deepgram only received ONE audio chunk (250ms) then got nothing for seconds.
  250ms of audio is not enough to transcribe anything.
  That is why "What you spoke" was always empty.

  Fix: Moved Groq call into the background transcript listener.
  Now the audio loop just does:
    while True:
        data = receive audio chunk
        send to Deepgram     <- nothing else, never blocked

  And separately, when Deepgram sends back a transcript:
    show transcript in "What you spoke"
    THEN call Groq with that transcript
    show Groq reply in "AI Response"

  This way Deepgram always gets continuous audio and can transcribe properly.


PROBLEM 7 - Groq was using a hardcoded message, not what you actually said
  Original code sent this to Groq every time, no matter what you said:
    "Explain the importance of fast language models"

  Fix: Now Groq receives the actual transcript of what you spoke:
    {"role": "user", "content": transcript}


PROBLEM 8 - Two outputs were mixed in one paragraph
  Original code sent both transcript and AI reply to the same <p> tag.
  You could not tell which was your speech and which was the AI.

  Fix: Added two separate sections in the HTML.
  Server prefixes every Groq chunk with "ai:" before sending.
  Browser checks the prefix and routes each message to the right section.


PROBLEM 9 - Deepgram connection was never closed on disconnect
  Original code had no cleanup when the browser disconnected.
  The Deepgram WebSocket connection was left open and leaked.

  Fix: The "async with" context manager closes Deepgram automatically.
  Also added a "finally" block to cancel the background listener task
  so it does not keep running after the browser disconnects.

      |

      v

      |

      v

      |
      | transcript (when speech detected)
      v
  Background listener task
      |
      |--- sends transcript --> Browser "What you spoke" section
      |
      |--- sends transcript to Groq AI
                |
                | streaming reply chunk by chunk
                v
            Browser "AI Response" section
